## d4068bb - Fix files explorer link generation to preserve path slashes

Date: 2026-05-07 22:14:39 -0400
Author: John Davenport

~p"/files/#{path}" URL-encodes "/" inside a string interpolation,
producing /files/marketing%2F08_plan.md. In dev, Plug.Static's
raise_on_missing_only path validator URI-decodes that and raises
InvalidPathError before the router ever sees the request.

Splitting the path into segments (~p"/files/#{String.split(path, "/")}")
uses the splat-route interpolation form, which preserves slashes
between segments without encoding them. Updates the matching spec
assertion to drop the URI.encode workaround.

## dc15db1 - Resolve OAuth sign-in by provider identity, not just email

Date: 2026-05-07 21:58:38 -0400
Author: John Davenport

Story 672/673 — Sign Up And Sign In With Google/GitHub.

The public OAuth sign-in flow (`UserOAuthController`) now resolves the
user by `(provider, provider_user_id)` first, falling back to email,
before creating a new account. Previously a Google primary-email change
created a duplicate MMS user (criterion 5681). Provider identity is
stored in `integrations.provider_metadata` (existing JSONB column) and
looked up via `Integrations.find_user_id_by_provider_identity/2`. The
controller also captures the OAuth token and upserts the integration
row, so re-sign-in refreshes tokens in place.

Side benefits: same email signing in via Google then GitHub now
consolidates onto one user record with both integrations attached.

Adds Google + GitHub buttons to `/users/register` (the original miss
that started this), and ports the ReqCassette-based OAuth test
infrastructure from code_my_spec (4 cassettes + `OAuthHelpers`) so the
8 spex in stories 672/673 can replay the full callback flow offline
instead of relying on `Req.Test.stub` (which doesn't intercept Assent).

Also fixes a wrong-route bug in 5680/5686: those denial-recovery spex
were exercising `/integrations/oauth/callback/:provider` (the
authenticated integration-add route) when the story is about public
sign-in via `/auth/:provider/callback`.

Result: 8/8 spex pass, no skips. Boundary widened to add `MarketMySpec`
to `MarketMySpecSpex`'s deps so helpers can seed `OAuthStateStore`.

## b6de16b - Migrate to SSM-bootstrap secrets pattern

Date: 2026-05-07 14:10:04 -0400
Author: John Davenport

Mirror metric_flow's pattern: container fetches its own secrets from
/<project>/<APP_ENV>/* at boot via <App>.Secrets.load!/1; Kamal carries
only AWS bootstrap creds + APP_ENV/AWS_REGION. Closes the operator-side
render-env exfiltration vector.

- lib/<app>/secrets.ex: paged GetParametersByPath + bounded retry,
  writes results into System env so the rest of runtime.exs sees them.
- config/runtime.exs: in :prod, calls <App>.Secrets.load!(app_env)
  before source!.
- mix.exs/mix.lock: ex_aws_ssm + hackney as direct deps; both in
  extra_applications so they boot before runtime.exs runs.
- .kamal/secrets-common: shrunk to KAMAL_REGISTRY_PASSWORD + AWS
  bootstrap creds only.
- config/deploy.yml (+ deploy.uat.yml where applicable): env.clear adds
  APP_ENV + AWS_REGION; env.secret only AWS_ACCESS_KEY_ID +
  AWS_SECRET_ACCESS_KEY.

App secrets at /<project>/<APP_ENV>/* in SSM are unchanged. The
bootstrap IAM users (<service>-{prod,uat}-app) have read-only access
to that path + kms:Decrypt on the SSM service.
