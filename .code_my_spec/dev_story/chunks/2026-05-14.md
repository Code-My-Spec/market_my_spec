## 9789030 - devops: migrate-before-swap + retry mix assets.deploy on transient 504s

Date: 2026-05-14 23:55:29 -0400
Author: John Davenport

Two papercuts from tonight's deploy:

**1. Pre-swap migrations.** Both deploy scripts now run
`kamal app exec --version=<sha> "/app/bin/migrate"` BEFORE
`kamal deploy`. The one-off container spins up on the new image,
runs the Phoenix release migration command, and exits. The currently-
serving container keeps handling traffic against the now-migrated
schema until the deploy step swaps in. Closes the gap where the
old "deploy → traffic swap → migrations forgotten" flow caused the
`column a0.subdomain does not exist` 500-storm on UAT tonight.

Caveat documented in the script: this assumes additive migrations.
A destructive change that drops a column the old code still reads
would break the old container during the migrate-before-swap
window. For that case, do the breaking change in two deploys
(stop reading the column → drop the column).

**2. Retry mix assets.deploy.** Wrap the Dockerfile's
`mix assets.deploy` step in a shell retry loop (up to 5 attempts,
linear backoff 5s/10s/15s/20s). Tonight's arm64 build failed because
github.com/tailwindlabs/tailwindcss/releases returned a 504 while the
tailwind elixir library was downloading the platform binary; the
amd64 build that ran concurrently succeeded with the same code. Retry
covers both the tailwind GitHub-releases endpoint and the esbuild
npm registry endpoint without needing to pre-fetch either binary
manually.

Files: Dockerfile, scripts/deploy, scripts/deploy-uat.

## 8694071 - Read apex_host at runtime so UAT + prod can share one Docker image

Date: 2026-05-14 23:28:18 -0400
Author: John Davenport

`MarketMySpec.Agencies.HostResolver.apex_host/0` was reading
`Application.compile_env(:market_my_spec, :apex_host, "marketmyspec.com")`
at compile time, so the BEAM built in CI hardcoded `marketmyspec.com`.
Booting that image on UAT (uat.marketmyspec.com) routed every request
into the agency-subdomain branch — `uat.marketmyspec.com` "ends in
.marketmyspec.com" — failed to resolve an agency named "uat", and 302
redirected the whole world back to https://marketmyspec.com. UAT was
DOA on first deploy.

- `host_resolver.ex`: drop the `@apex_host` module attribute, read
  `Application.get_env/3` at call time inside `apex_host/0`, and call
  `apex_host()` from `extract_subdomain/1`.
- `runtime.exs`: set `config :market_my_spec, :apex_host, host` after
  reading `PHX_HOST`. UAT lands on `uat.marketmyspec.com`; prod on
  `marketmyspec.com`. Agency subdomains on UAT become
  `<agency>.uat.marketmyspec.com`.

168/168 mix tests stay green.

## 6b56f6e - Add saved-search admin (story 710) — schemas, repo, MCP-shaped LiveView

Date: 2026-05-14 22:46:41 -0400
Author: John Davenport

Solo founders can save named keyword-list searches scoped to a chosen
subset of their venues, then re-run by name from either the LiveView
admin or the MCP agent. Saved searches are recipes only — no run
history persisted.

Schema (single Google-style query string, NOT a keyword list):

- MarketMySpec.Engagements.SavedSearch — `name` (unique per account),
  `query` (single string supporting "quoted phrases" / AND / OR /
  negation, parsed at run time), `source_wildcards` (array of
  source atoms, lets a search say "all enabled ElixirForum venues"
  without enumerating each one), many-to-many with Venue.
- MarketMySpec.Engagements.SavedSearchVenue — join table; account_id
  denormalized for fast account-scoped queries and so cross-account
  venue links are rejected at insert.
- Migration 20260514100000 creates both tables.
- Migration 20260514110000 brings drifted dev DBs onto the agreed
  shape without `mix ecto.reset` (forward-only `IF EXISTS`/`IF NOT
  EXISTS` alignment). The 20260514100000 file was rewritten in place
  during a design rollback; envs that recorded the old version
  skipped the rewrite, so this corrective migration adds the missing
  columns and drops the legacy ones.

Repository: MarketMySpec.Engagements.SavedSearchesRepository

- list/get/create/update/delete + run_saved_search/2.
- create/update validate that every venue_id in attrs belongs to the
  caller's account before insert.
- "at least one selector" rule enforced at the repo: either a
  non-empty venue_ids OR a non-empty source_wildcards.
- run_saved_search resolves explicit venues + wildcard-expanded
  enabled venues per source, dedupes by id, and delegates to
  Engagements.Search.search/3. Zero resolved venues → empty
  candidates list, not an error.
- All queries scope on `current_scope.active_account_id`;
  cross-account access returns `:not_found`.

LiveView: MarketMySpecWeb.SearchLive.Index at /accounts/:id/searches

- Account-scoped admin patterned on VenueLive.Index. Inline
  create/edit form (name, single query text input, venue multi-select,
  per-source wildcard checkboxes). Per-row "Run now" renders the
  candidate list inline. Cross-account URL access redirects with
  "Account not found".
- Route registered in router.ex under :require_authenticated_user.

10 BDD spex (criteria 6227-6236) covering venue selectors,
wildcard expansion, name uniqueness (per-account, mutable),
Google-syntax query persistence, account scoping, and the

## 2a737ab - Add Engagements feature — search, thread fetch, stage response, venue admin

Date: 2026-05-14 12:04:08 -0400
Author: John Davenport

Implements stories 705/706/707/708 of the engagement-finder loop. The LLM
agent (over MCP) and the user (via LiveView admin) both manage venues per
source, run searches, ingest threads, and stage UTM-embedded draft responses.

**Domain (`MarketMySpec.Engagements`):**
- `Venue` schema — per-account, source-typed identifier (subreddit name |
  category + optional tag), weight, enabled flag; per-source identifier
  validation via the Source behaviour.
- `Thread` schema — account-scoped ingested thread with normalized OP body,
  comment_tree, and raw_payload preserved for re-render.
- `Touchpoint` schema — staged or posted comment record; comment_url and
  posted_at are nullable so staged drafts can exist before posting.
- `Source` behaviour with Reddit and ElixirForum adapters. v1 is read-only:
  both `post/3` callbacks return `{:error, :posting_not_supported}`. Users
  copy-and-paste polished drafts manually.
- Repositories (`VenuesRepository`, `ThreadsRepository`, `TouchpointsRepository`)
  enforce `Scope.active_account_id` scoping — cross-account access returns
  `:not_found`.
- `Search` orchestrator fans out across enabled venues in parallel
  (`Task.async_stream`), deduplicates by URL, ranks by `venue.weight ×
  per_source_signal`, and reports per-source failures without crashing the
  call.
- `Posting.build_utm_url/2` embeds per-source UTM scheme
  (`utm_source=reddit|elixirforum`, `utm_medium=engagement`,
  `utm_content=<thread_id>`) into the user-provided link_target before the
  staged Touchpoint is persisted.

**MCP tools:** `search_engagements`, `get_thread`, `stage_response`. All
three are registered in `MarketingStrategyServer` so LLM clients see them
in `tools/list`.

**LiveView surfaces:**
- `VenueLive.Index` — account-scoped CRUD admin at `/accounts/:id/venues`
  with enabled toggle, weight edit, per-source identifier validation.
- `ThreadLive.Index` and `ThreadLive.Show` — list and per-thread views at
  `/accounts/:id/threads` and `/accounts/:account_id/threads/:thread_id`
  rendering the normalized OP + comment tree alongside any staged or
  posted Touchpoint, with the embedded UTM link visible and copy-to-clipboard
  / mark-posted actions.

**Migrations:** venues + threads + touchpoints tables with account_id +
unique constraints; touchpoint comment_url/posted_at made nullable to
support staged-before-posted state.

**Tests:** 75 BDD spex files across stories 705 (18), 706 (12), 707 (12),
708 (25), plus a Repo-backed test for `TouchpointsRepository`. Fixtures
extended with engagement-side helpers. All 249 spex pass; full `mix test`
suite green.

**Config:** `config/test.exs` flips the Endpoint `:server` to `false` —
current spex run in-process via LiveViewTest; flip back to `true` only
when a Wallaby feature test needs a real browser.

## 0d32dd5 - Block mismatched signed-in users from accepting invitations (story 696)

Date: 2026-05-14 12:03:06 -0400
Author: John Davenport

The accept LiveView's `handle_event("accept_invitation")` always passed
`%{email: invitation.email}` to `Accounts.accept_invitation/2`, so the
`:email_mismatch` guard in `Accounts.resolve_or_create_user/2` was unreachable
from the browser. A signed-in user could accept an invitation addressed to a
different email — silently joining the account as the wrong user.

- `mount/3` now derives `mismatched_user` from `current_scope.user.email` vs
  `invitation.email` (false when not signed in).
- Template surfaces a "Wrong account" warning with a sign-out link and disables
  both accept buttons via `disabled={@mismatched_user}`.
- `handle_event` short-circuits with a flash error when `mismatched_user` is
  true — defense in depth against a forged accept event with the button
  disabled bypassed.
- Beef up the 6117 spex per `feedback_beef_up_anemic_spex.md`: the previous
  spec only checked rendered HTML for the not-yet-an-account invitee branch
  (passes vacuously against buggy code). Added a scenario reproducing the
  exact QA failure mode (existing-user invitee, mismatched signed-in user
  clicks Accept) asserting the invitation stays `:pending` and no membership
  is created. Verified the new spec correctly fails against unpatched code.

invitation management UI the story implements. Routes `/accounts/:id/invitations`
swap from the placeholder AccountLive.Invitations to InvitationsLive.Index.
