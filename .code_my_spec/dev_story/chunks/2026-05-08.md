## db15534 - Pin active_account_id in user_fixture for deterministic scoping

Date: 2026-05-08 13:27:45 -0400
Author: John Davenport

Tests that create a second account after `user_fixture()` were flaky
because `Scope.for_user/1` picks the user's `active_account_id` when
set, otherwise the "newest by inserted_at" account. With two accounts
inserted within the same DB tick, the order tied and Postgres returned
rows in arbitrary sequence — sometimes the default-individual account
was active, sometimes the second one was.

Story 684 spex 5984/5985/5988 hit this every time the active account
flipped to "B" while the test assumed "A" was active, producing leaked
files in tree assertions.

Setting active_account_id on the user struct after the default account
is created makes Scope.for_user deterministic. Five seed-0 runs of the
full spex suite are clean.

## 2da53c0 - Drop deprecated Appsignal use clause and purge ETS between spex

Date: 2026-05-08 13:23:28 -0400
Author: John Davenport

- Remove `use Appsignal.Phoenix` from endpoint — Phoenix instrumentation
  has been automatic since AppSignal 2.3.0; the call now warns at compile.
- Purge `:market_my_spec_files_memory` in MarketMySpecSpex.Case setup so
  files written by one spex don't leak into the next. The Memory adapter
  uses a single named ETS table that survives the SQL sandbox rollback;
  without this, criteria 5984/5985/5988 in story 684 flake on order. This
  doesn't fix every intra-test ordering issue in 684, but cuts most of
  the cross-test leakage.
