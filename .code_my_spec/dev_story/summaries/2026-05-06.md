# 2026-05-06

The day was a focused deploy-infrastructure session, untangling Kamal's destination model so prod and UAT could finally live as separate environments off the same image. Four commits, all landing within a half-hour window late morning, each one fixing a specific friction point in the multi-environment pipeline.

The work opened with the architectural pivot: multi-arch image builds. The GitHub Actions workflow learned to emit both `linux/amd64` and `linux/arm64` variants via QEMU, and `config/deploy.uat.yml` was repointed at the German Hetzner cax11 ARM box in Nuremberg (46.225.105.88), peeling UAT off the prod host for the first time. Prod stayed x86, UAT went ARM, and both pulled the matching variant from a single shared manifest - a clean split that avoided maintaining two build pipelines.

A small but load-bearing config fix followed: `proxy: false` was added to the prod role to mirror UAT. The setting had been working in the live deploy state but uncommitted, so the commit was less a change and more a lockdown for reproducibility.

Then came two Kamal idiom corrections. First, `.kamal/secrets` was renamed to `.kamal/secrets-common` - Kamal's convention is per-destination secret files with `secrets-common` as the shared fallback, and the old name had silently only worked for the default destination, leaving UAT unable to find its credentials. Second, the service name was unified to `market-my-spec` across destinations after the realization that Kamal expects one service per project, with the destination label doing the disambiguation. Distinct service names per destination had been causing each `.uat` config to look for an image label that didn't match what GHA was building.

A short, surgical infra day - no application code touched, just the four small commits needed to make the prod/UAT split actually work end-to-end.
