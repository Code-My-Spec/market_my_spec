## Issues

### Migration file re-written without rolling back existing table causes schema drift

#### Severity
HIGH

#### Scope
APP

#### Description
The migration `priv/repo/migrations/20260514100000_create_saved_searches.exs` was re-written to use the new schema (`query` string + `source_wildcards` array + many-to-many `saved_search_venues`), but the dev database already had a `saved_searches` table from a previous version of this migration file that used the old schema (`keywords` array, `venue_identifiers` array, `venue_source` string). Because `schema_migrations` already recorded version `20260514100000` as run, `mix ecto.migrate` reports "Migrations already up" and does not apply the updated migration content.

The result is that the running app (after the server restart) throws `Postgrex.Error: ERROR 42703 (undefined_column) column s0.query does not exist` on every request to `/accounts/:id/searches`, making the entire LiveView surface non-functional.

To reproduce on a dev environment that has already run an older version of migration `20260514100000`:

1. `mix ecto.migrate` — reports "already up"
2. Navigate to `/accounts/<id>/searches` as authenticated user
3. Observe Postgrex error in the browser

The fix applied during this QA session was a manual ALTER TABLE to drop old columns and add new ones. The proper fix is to either: (a) roll back to before `20260514100000` and re-run `mix ecto.reset` on affected dev environments, or (b) create a new migration that performs the column rename/add/drop explicitly.

Any dev environment that ran the original version of `20260514100000` will have this issue on `mix ecto.migrate`. Fresh environments or environments that ran `mix ecto.reset` after the migration was updated will be fine.

### /accounts/:id/searches route not registered in running server without restart

#### Severity
MEDIUM

#### Scope
APP

#### Description
The route `live "/accounts/:id/searches", SearchLive.Index, :index` was present in `router.ex` but the running Phoenix dev server did not pick it up via code reload (`SyncCodeReloadPlug`) — requests to the URL returned `Phoenix.Router.NoRouteError`. A server restart was required to register the route.

This is consistent with Phoenix behavior where router changes sometimes require a full server restart rather than being picked up by the dev code reloader. However, it indicates the route was added to the source file but may not have been in place when the dev server was started for this QA session.

Reproduction: run `PORT=4007 mix phx.server` before the router.ex change is compiled, then modify router.ex — the new route may not appear until server restart.

The issue was resolved by restarting the server with `PORT=4007 mix phx.server`. After restart, `mix phx.routes` confirmed the route is present and the URL returned 200.
