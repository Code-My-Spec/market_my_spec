## Issues

### McpSetupLive missing data-test attributes on step elements (install-step, oauth-step, interview-step)

#### Severity
HIGH

#### Description
The BDD spec (criterion 5705) asserts `[data-test='install-step']`, `[data-test='oauth-step']`, and `[data-test='interview-step']` on the three `<li>` step elements. The current `McpSetupLive` render in `lib/market_my_spec_web/live/mcp_setup_live.ex` has no `data-test` attribute on the install `<li>`, uses `data-test='oauth-instructions'` (not `oauth-step`) on the OAuth `<li>`, and has no `data-test` attribute on the interview `<li>`.

Reproduction: authenticate as `qa@marketmyspec.test` and visit `http://localhost:4008/mcp-setup`. Inspect the `<ol class="steps steps-vertical">` element — the child `<li>` elements lack the required `data-test` attributes.

### McpSetupLive missing expected-result verification element

#### Severity
HIGH

#### Description
Criterion 5706 requires `[data-test='expected-result']` containing text matching `connected`, `success`, `working`, or `installed`. This element does not exist anywhere on the `/mcp-setup` page. The acceptance criteria explicitly states: "Page lacking expected-result verification step is rejected." A user following the guide has no way to verify that their setup succeeded.

Reproduction: authenticate and visit `http://localhost:4008/mcp-setup`. No `[data-test='expected-result']` element exists on the page.

### McpSetupLive missing all three required troubleshooting blocks

#### Severity
HIGH

#### Description
Criteria 5707 and 5708 require three troubleshooting blocks with `data-test` attributes:
- `[data-test='port-conflict-troubleshooting']` — guidance for port conflict errors, must mention "port"
- `[data-test='oauth-troubleshooting']` — guidance for OAuth authorization failures
- `[data-test='mcp-connection-troubleshooting']` — guidance for MCP connection issues

None of these blocks exist in the current `McpSetupLive` implementation. The acceptance criteria explicitly states: "Page missing one of the three required troubleshooting blocks is rejected."

Reproduction: authenticate and visit `http://localhost:4008/mcp-setup`. The page has only the three-step guide. No troubleshooting section exists.

### QA plan documents port 4007 but active server with current code is on port 4008

#### Severity
LOW

#### Scope
QA

#### Description
The QA plan at `.code_my_spec/qa/plan.md` documents the base URL as `http://localhost:4007`. The seed script prints magic-link URLs with port 4007. During this session, the server on port 4007 was stale (returned 404 for `/mcp-setup`) while port 4008 served the current code (returned 302). The magic-link tokens are stored in the database and work regardless of port, but testers need to manually substitute 4008 for 4007 in the URL. The QA plan's System Issues section should be updated to document this port discrepancy and the workaround (use 4008 for the current session, or kill the stale 4007 process and restart on the canonical port).
