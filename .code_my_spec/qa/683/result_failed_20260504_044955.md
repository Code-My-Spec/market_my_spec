### 6. Static Audit: Path Traversal and Absolute Path Rejection

Pass.

`lib/market_my_spec/files.ex` `validate_path/1` correctly rejects:
- Paths beginning with `/` — returns `{:error, :invalid_path}`
- Paths containing `..` anywhere — returns `{:error, :invalid_path}`

The implementation is:

```elixir
defp validate_path("/" <> _), do: {:error, :invalid_path}
defp validate_path(path) do
  case String.contains?(path, "..") do
    true -> {:error, :invalid_path}
    false -> :ok
  end
end
```

This is called from `resolve/2` before any backend delegation, so both ReadFile, WriteFile, and ListFiles inherit the rejection without additional code. When EditFile and DeleteFile are implemented (delegating through `Files.get/2`, `Files.put/3`, `Files.delete/2`), they will inherit the same protection.

### 7. Static Audit: Cross-Tenant Scoping

Pass.

`resolve/2` in `lib/market_my_spec/files.ex` applies `accounts/{account_id}/` as a server-side prefix to every path before backend delegation:

```elixir
defp account_prefix(%Scope{active_account_id: id}), do: "#{@account_root}/#{id}/"
```

`list/2` strips the account prefix from returned entries via `strip_prefix/2`, so callers only see account-relative keys. `resolve/2` returns `{:error, :no_active_account}` when `active_account_id` is nil, preventing unauthenticated writes or reads.

### 8. Static Audit: Write-Before-Overwrite Gating (Session Gate)

Fail.

The story requires that `write_file` on an existing path must reject the call unless the caller has previously called `read_file` on that path in the same MCP session. The current `WriteFile.execute/2` delegates directly to `Files.put/3` with no session-state check:

```elixir
def execute(%{path: path, content: content}, frame) do
  scope = frame.assigns.current_scope
  case Files.put(scope, path, content) do
    {:ok, _metadata} -> ...
```

There is no inspection of `frame.context.session_id` or any session-keyed read-set. Overwrite gating is completely absent. Spex criterion 5837, 5860, and 5861 all assert this gate exists and will fail when run.

The same gating requirement applies to edit_file and delete_file (which also do not exist yet).

## Issues

### EditFile and DeleteFile tool modules are missing

#### Severity
HIGH

#### Scope
APP

#### Description
The story requires five MCP file tools: read_file, write_file, edit_file, delete_file, list_files. Only three are implemented: ReadFile, WriteFile, ListFiles. The modules `MarketMySpec.McpServers.Marketing.Tools.EditFile` and `MarketMySpec.McpServers.Marketing.Tools.DeleteFile` do not exist anywhere in the codebase.

Affected acceptance criteria: edit_file exact-string replacement (5838), delete_file removes object (5839), edit_file replaces unique old_string (5862), edit_file with replace_all (5863), edit_file without prior read rejected (5864), edit_file non-unique old_string rejected (5865), edit_file on missing path returns not_found (5866), delete_file after read removes object (5867), delete_file without prior read rejected (5868).

Additionally, EditFile and DeleteFile are not registered in `MarketingStrategyServer`, meaning they would not appear in the `tools/list` response even if created, until added via `component(...)`.

### MarketMySpec.McpServers.Marketing module with tools/0 function is missing

#### Severity
HIGH

#### Scope
APP

#### Description
Four spex files (criteria 5834, 5842, 5851, 5852) call `MarketMySpec.McpServers.Marketing.tools()`. This module does not exist. The MCP server module is `MarketMySpec.McpServers.MarketingStrategyServer`, which does not expose a public `tools/0` function.

The spex need a `Marketing` facade module (or the spex need to be updated to call `MarketingStrategyServer.__tools__()` or equivalent Anubis introspection API). Until this is resolved, at least 4 spex will fail to compile.

### write_file does not enforce read-before-overwrite gating

#### Severity
HIGH

#### Scope
APP

#### Description
The story acceptance criteria state: "write_file with a path that already exists requires the caller to have read_file'd that path earlier in the same MCP session; without a prior read, the call returns a read-required error and does not overwrite."

The current `WriteFile.execute/2` calls `Files.put/3` unconditionally with no session state check. There is no mechanism to track which paths have been read in a given MCP session, and no gate that returns a read-required error on overwrite attempts.

Spex criteria 5837, 5860, and 5861 exercise this gate directly and will fail. The same session-gating requirement applies to EditFile and DeleteFile when implemented.

To fix: WriteFile needs to check a session-keyed read set (stored in `frame.context` or a process-keyed ETS table) before delegating to `Files.put/3` when the path already exists. ReadFile needs to add the path to that read set on success.

### Files context has no edit function for exact-string replacement

#### Severity
HIGH

#### Scope
APP

#### Description
The story requires `edit_file` to perform exact-string replacement of `old_string` with `new_string`, reject non-unique matches unless `replace_all` is true, and return not_found for missing paths. The Files context (`lib/market_my_spec/files.ex`) has no `edit/4` function or equivalent.

When EditFile is implemented, it will need to either: (a) call `Files.get/2` to fetch the body, do string replacement logic in the tool module, then call `Files.put/3`, or (b) add a `Files.edit/4` function to encapsulate this. The uniqueness check and replace_all branching are non-trivial and must be implemented somewhere.

This is a design gap, not just a missing module — the implementation path needs to be decided.
