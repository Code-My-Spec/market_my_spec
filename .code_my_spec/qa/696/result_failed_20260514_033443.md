## Issues

### Signed-in mismatched user can accept invitations meant for others

#### Severity
HIGH

#### Description
When a user is signed in and navigates to an invitation acceptance URL that was sent to a
DIFFERENT email address, they are not blocked. The accept page's `handle_event("accept_invitation")`
always passes `%{email: invitation.email}` to `Accounts.accept_invitation/2` regardless of the
current signed-in user's identity.

Reproduction:
1. Log in as `qa-owner@marketmyspec.test`
2. Navigate to `/invitations/accept/<token>` where the token belongs to an invitation for
   `qa-existing@marketmyspec.test`
3. The page shows "Welcome back!" (for the INVITEE, not the signed-in user)
4. Click "Accept Invitation"
5. The invitation is accepted and `qa-existing@marketmyspec.test` is added as a member
6. Flash says "Invitation accepted successfully!" — no error or block

Root cause in `lib/market_my_spec_web/invitations_live/accept.ex`:
- `handle_event("accept_invitation")` builds `user_attrs = %{email: socket.assigns.invitation.email}`
- This hardcodes the email to the invitation's email, not `current_scope.user.email`
- The `{:error, :email_mismatch}` error path in `Accounts.resolve_or_create_user/2` is unreachable
  through the normal browser flow

Expected behavior: If a user is signed in and their email does NOT match the invitation email,
the page should show an error or prevent acceptance.

### Invitation emails contain wrong port in accept URL

#### Severity
LOW

#### Scope
QA

#### Description
Invitation emails rendered in the dev mailbox (`/dev/mailbox`) show accept URLs with port 4000
(e.g., `http://localhost:4000/invitations/accept/<token>`) instead of the running server's
port 4008.

The app runs on port 4008 in dev, but the URL helper in `InvitationNotifier` (or the Endpoint
configuration) generates URLs with port 4000. Links in the mailbox fail to navigate correctly
until the port is manually corrected.

This blocks testing accept flows from the mailbox link directly. A tester must manually change
4000 to 4008 in the URL or use the token from the email body.

Workaround: copy the token from the email text body and construct the accept URL manually:
`http://localhost:4008/invitations/accept/<token>`

### Seed script creates invitations via separate BEAM process not captured in Swoosh mailbox

#### Severity
LOW

#### Scope
QA

#### Description
When `mix run priv/repo/qa_seeds_696.exs` creates invitations via `Accounts.invite_user/4`,
the invitation emails are sent through the BEAM process running the mix task, not through the
Phoenix server's Swoosh dev adapter. As a result, these invitation tokens do NOT appear in
the `/dev/mailbox` mailbox preview.

To get invitation tokens for browser-based acceptance testing, the tester must:
1. Cancel the seed-created invitation via the browser UI
2. Re-invite using the browser UI invite form
3. Then retrieve the token from `/dev/mailbox`

A future improvement would be for the seed script to print the raw invitation tokens directly,
since they are available before insert as the encoded token.
