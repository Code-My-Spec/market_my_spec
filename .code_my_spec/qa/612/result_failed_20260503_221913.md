## Scenarios

### Scenario 1: RFC 8414 well-known authorization server metadata (criterion 5691, 5692)

fail

Probe: `GET http://localhost:4008/.well-known/oauth-authorization-server`

Result: `404 Not Found` — Phoenix raises `NoRouteError`. The route is not registered in the router. The `McpAuth.ConnectionInfo` module exists and can produce the correct metadata map, but there is no `OauthController` or route to expose it at `/.well-known/oauth-authorization-server`.

Both criterion 5691 (MCP client auto-discovers endpoints via well-known metadata) and 5692 (metadata document completeness) cannot be satisfied — the endpoint does not exist.

fail

## Issues

### OauthController and OAuth routes are not implemented

#### Severity
CRITICAL

#### Scope
APP

#### Description
The router has no routes for any of the OAuth endpoints required by this story:

- `GET /.well-known/oauth-authorization-server` (RFC 8414)
- `GET /.well-known/oauth-protected-resource` (RFC 9728)
- `POST /oauth/register` (dynamic client registration)
- `GET /oauth/authorize` (authorization endpoint)
- `POST /oauth/token` (token endpoint)
- `POST /oauth/revoke` (revocation endpoint)

The `MarketMySpec.McpAuth` context (Token, Authorization, ConnectionInfo) is fully implemented and ready to be wired up. The `McpAuth.ConnectionInfo` module produces correct RFC 8414 and RFC 9728 metadata maps. A reference `OauthController` implementation exists at `/Users/johndavenport/Documents/github/code_my_spec/lib/code_my_spec_web/controllers/oauth_controller.ex`.

Reproduction: `curl -o /dev/null -w "%{http_code}" http://localhost:4008/.well-known/oauth-authorization-server` returns 404.

All 12 acceptance criteria for story 612 fail because of this missing controller + router wiring.

### McpController is not implemented — /mcp endpoint returns 404

#### Severity
CRITICAL

#### Scope
APP

#### Description
The `/mcp` JSON-RPC endpoint does not exist. Probing `POST http://localhost:4008/mcp` returns 404 for both authenticated and unauthenticated requests.

Three acceptance criteria directly test MCP endpoint behavior (criterion 5693 valid bearer grants access, criterion 5694 expired token returns 401, criterion 5701 revoked token rejected, criterion 5703 unauthenticated returns 401 with auth pointer). All are blocked.

Story context notes McpController as pending in the requirement graph.

### McpAuthorizationLive is not mounted in the router

#### Severity
HIGH

#### Scope
APP

#### Description
The `McpAuthorizationLive` module (`lib/market_my_spec_web/live/mcp_authorization_live.ex`) is implemented but no route in `router.ex` mounts it. Navigating to `/oauth/authorize` (where the OauthController would redirect for user consent) and `/mcp/authorize` both return `NoRouteError`.

The LiveView needs to be mounted at the path the `OauthController` will redirect to for user consent — typically `/oauth/authorize` as a `live` route inside an authenticated `live_session`.

### McpAuthorizationLive approve button is missing data-test attribute

#### Severity
HIGH

#### Scope
APP

#### Description
The approve button in `McpAuthorizationLive` renders as:

```elixir
<.button phx-click="approve" class="btn-primary">Approve</.button>
```

The BDD specs for criteria 5689, 5690, 5693, and 5701 all select this button with:

```elixir
element("[data-test='approve-button']")
```

Without the `data-test="approve-button"` attribute, all four BDD specs will fail the `render_click()` step with an "element not found" error even once the route is mounted correctly.

Fix: add `data-test="approve-button"` to the approve button's attributes.
