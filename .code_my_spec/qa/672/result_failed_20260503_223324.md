## Status

fail

## Scenarios

### Scenario 1: Google sign-in button present on login page (Criterion 5679)

fail

Navigated to `http://localhost:4008/users/log-in`. The page renders a magic-link email form and a password login form. There is no element with `data-test="google-sign-in"` on the page. Confirmed with `browser_is_visible("[data-test='google-sign-in']")` returning false, and by reading the full page text — no Google sign-in option appears anywhere on the login page.

The BDD spec (criterion_5679_spex.exs) asserts `has_element?(view, "[data-test='google-sign-in']")` on the `/users/log-in` page. This assertion would fail against the current implementation.

Evidence: `.code_my_spec/qa/672/screenshots/672-login-page.png`

### Scenario 2: OAuth redirect to Google initiates correctly (Criterion 5679)

fail

After signing in via magic-link and navigating to `/integrations`, both GitHub and Google appear in the "Available" section with Connect links. Clicked the Google Connect link (which navigates to `/integrations/oauth/google`).

The request crashed with a RuntimeError: "default CA trust store not available; please add `:castore` to your project's dependencies." The `castore` package is not listed in `mix.exs`. The Assent library requires `:castore` for SSL certificate verification when making HTTPS calls to Google's OIDC configuration endpoint. The app raised a 500 error instead of redirecting to Google's authorization endpoint.

This is a crash in the OAuth request flow — the `IntegrationsController.request/2` function does not rescue the error, and the Plug/Bandit stack surfaces it as a debug error page.

Evidence: `.code_my_spec/qa/672/screenshots/672-google-oauth-error.png`, `.code_my_spec/qa/672/screenshots/672-integrations-page.png`

### Scenario 3: Access denied callback redirects with error flash (Criterion 5680)

pass

While authenticated as `qa@marketmyspec.test`, navigated to `/integrations/oauth/callback/google?error=access_denied`. The controller's `callback/2` function detected the `error` param, called `format_oauth_error("access_denied", params)`, and redirected to `/integrations` with the flash message: "You denied access. Please try again if you want to connect."

The flash message contains "denied", satisfying `~r/denied|access/i`. The redirect destination is `/integrations`, matching the expected `redirected_to =~ "/integrations"`. The flash message is present and correct.

Evidence: `.code_my_spec/qa/672/screenshots/672-access-denied-flash.png`

### Scenario 4: Recovery after denied consent (Criterion 5680)

pass

After the access_denied callback redirect landed on `/integrations`, the page rendered normally. Google appeared in the "Available" section with a Connect button, confirming the user can attempt the OAuth flow again. No stale state or broken UI detected.

Evidence: `.code_my_spec/qa/672/screenshots/672-access-denied-callback.png`

### Scenario 5: Missing sub claim is rejected (Criterion 5682 — source code review)

pass (code review)

Live testing with a mocked Google token response requires test stubs (`Req.Test.stub/2`) which cannot be used in browser automation. Source code review confirms correct behavior:

In `lib/market_my_spec/integrations/providers/google.ex`, `extract_provider_user_id/1` pattern-matches on `%{"sub" => nil}` and `_user_data` (no sub key) returning `{:error, :missing_provider_user_id}`. This propagates through `normalize_user/1` returning `{:error, :missing_provider_user_id}`.

In `lib/market_my_spec/integrations.ex`, `handle_callback/6` pattern-matches the `{:error, reason}` branch. In `lib/market_my_spec_web/controllers/integrations_controller.ex`, `callback/2` catches this and sets `put_flash(:error, "Failed to complete connection")` and redirects to `/integrations`. The flash message matches `~r/failed|error/i`. No integration record is created in the error path.

### Scenario 6: Email change resolves to same account via sub (Criterion 5681 — source code review)

pass (code review)

Live testing requires mocked Google HTTP responses. Source code review confirms correct behavior:

The `upsert_integration/3` function in `IntegrationRepository` uses `conflict_target: [:user_id, :provider]` — the integration record is keyed on `(user_id, provider)`, not on email. When Google returns a callback with the same `sub` but a different email, the `normalize_user/1` extracts `sub` as `provider_user_id` (stored in `provider_metadata`), and the upsert updates the existing row. The email field is stored in `provider_metadata` and would be updated, but the record identity (and user linkage) is unchanged.

The `sub` claim serves as the stable account identifier as required by criterion 5681.

## Issues

### Google sign-in button missing from login page

#### Severity
HIGH

#### Scope
APP

#### Description
The `/users/log-in` page does not have a Google sign-in option. The BDD spec (criterion_5679_spex.exs) asserts `has_element?(view, "[data-test='google-sign-in']")` on the login page, but the login page (`lib/market_my_spec_web/live/user_live/login.ex`) only renders a magic-link form and a password form — no OAuth sign-in buttons. The acceptance criterion "New visitor signs up via Google in one click" requires Google sign-in to be accessible from the login page without first creating an account and navigating to /integrations.

Reproduction: visit `http://localhost:4008/users/log-in` as an anonymous user. No Google sign-in option is present.

### Google OAuth flow crashes with missing castore dependency

#### Severity
HIGH

#### Scope
APP

#### Description
Clicking the Google Connect button at `/integrations/oauth/google` raises a RuntimeError: "default CA trust store not available; please add `:castore` to your project's dependencies." The `:castore` package is not listed in `mix.exs`. Without it, Assent (via Req/Finch/Mint) cannot establish SSL connections to Google's OIDC configuration endpoint (`accounts.google.com`).

The crash surfaces as an unhandled exception in `IntegrationsController.request/2` — the controller does not rescue SSL/transport errors from `Integrations.authorize_url/3`. Users see a 500 debug error page instead of a redirect to Google or a graceful error message.

Fix: add `{:castore, ">= 0.0.0"}` to `deps` in `mix.exs` and run `mix deps.get`. Additionally, consider adding a rescue around `authorize_url/3` in the controller to return a user-facing flash error instead of a 500 crash if credentials or transport fail.

Reproduction: sign in as any user, navigate to `/integrations`, click Google Connect.

### QA plan has stale port (4007 vs 4008)

#### Severity
LOW

#### Scope
QA

#### Description
The QA plan at `.code_my_spec/qa/plan.md` documents the app as running on port 4007. During this session, 4007 was not responding and 4008 was the active port. The seed script's magic-link URL also hardcodes port 4007, requiring manual substitution. The plan's System Issues section documents the dotenvy PORT issue but does not update the canonical port or note that 4008 may be used instead.

Workaround used: manually replaced 4007 with 4008 in all URLs. Re-ran seeds and substituted port in the printed magic-link URL.
