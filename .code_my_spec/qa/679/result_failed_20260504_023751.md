## Issues

### Agency dashboard route mounted at /agency/dashboard instead of /agency

#### Severity
HIGH

#### Description
The router mounts `AgencyLive.Dashboard` at `/agency/dashboard` (line 84 of `router.ex`), but all 13 BDD spex files navigate to `/agency`. This causes every spex test that uses `live(conn, "/agency")` to get a 404 response and raise `FunctionClauseError` in `Phoenix.LiveViewTest.connect_from_static_token/3`. The route must be changed to `/agency` in the router, or all spex files must be updated to use `/agency/dashboard`. Given the spec contract uses `/agency`, the router is the correct place to fix this.

### Individual-account users can access the agency dashboard

#### Severity
HIGH

#### Description
Navigating to `http://localhost:4008/agency/dashboard` as a user with an individual-typed account renders the "Client Dashboard" page without any redirect or error. Criterion 5782 requires that individual-account users be redirected away from `/agency` to `/accounts` or `/`. The `AgencyLive.Dashboard.mount/3` performs no account type check ‚Äî it unconditionally assigns `client_accounts: []` and renders. A guard must be added to mount (or via a custom on_mount hook) that checks `current_scope.active_account.type` and redirects individual-account users away.

### Missing data-test attributes on the agency dashboard

#### Severity
HIGH

#### Description
The `AgencyLive.Dashboard` template (`lib/market_my_spec_web/live/agency_live/dashboard.ex`) has no `data-test` attributes on any element. The BDD specs require:
- `[data-test='agency-client-dashboard']` on the container (criterion 5781)
- `[data-test='client-row']` on each table row (criteria 5791, 5795, 5788, 5789, 5792)
- `[data-test='client-name']` on the account name cell (criterion 5791)
- `[data-test='access-level']` on the access level cell (criterion 5791)
- `[data-test='client-row-originator']` attribute on originator rows (criteria 5783, 5784)
- `[data-test='client-row-invited']` attribute on invited rows (criteria 5785, 5786)
- `[data-test='enter-client']` on the enter button (criteria 5788, 5789, 5792)
- `[data-test='revoke-grant']` on revoke controls for invited rows (criteria 5784, 5786)

### Agency dashboard mount does not query or populate client accounts

#### Severity
HIGH

#### Description
`AgencyLive.Dashboard.mount/3` hardcodes `assign(socket, :client_accounts, [])`. It never queries the database for the agency's client accounts. To satisfy criteria 5781, 5783, 5785, 5791, 5795, the mount must query for all agency-client access grants associated with the current scope's active agency account and populate the `:client_accounts` assign with the result.

### Missing fixtures: account_fixture, invited_grant_fixture, originated_client_fixture, account_member_fixture

#### Severity
HIGH

#### Scope
QA

#### Description
Multiple BDD spex files call fixture functions that do not exist in `MarketMySpecSpex.Fixtures` or `MarketMySpec.UsersFixtures`:

- `Fixtures.account_fixture/2` ‚Äî needed by criteria 5786, 5789, 5790, 5791, 5795
- `Fixtures.invited_grant_fixture/3` ‚Äî needed by criteria 5785, 5786, 5789, 5790, 5791, 5795
- `Fixtures.originated_client_fixture/2` ‚Äî needed by criteria 5795, 5796
- `Fixtures.account_member_fixture/3` ‚Äî needed by criterion 5792

These cause `UndefinedFunctionError` at the `given_` step before any LiveView navigation occurs. The fixtures must be implemented and delegated from `MarketMySpecSpex.Fixtures` before the corresponding criteria can run. This also implies missing domain-layer infrastructure: there is no `AgencyClientGrant` schema, no migration for an `agency_client_grants` table, and no context function for creating or querying grants.

### Missing /agency/clients/new route and form

#### Severity
HIGH

#### Description
Criteria 5783 and 5784 require that an agency owner can navigate to `/agency/clients/new` and submit a `[data-test='client-form']` to create a client account. This route does not exist in the router ‚Äî it returns 404. The route and a corresponding LiveView or form component must be implemented.

### Missing grant-agency-access-form on /accounts page

#### Severity
HIGH

#### Description
Criterion 5785 requires that a client owner can grant an agency access from the `/accounts` page via a `[data-test='grant-agency-access-form']` form. No such form exists in `AccountLive.Index`. This form (with fields for agency slug and access level) must be added to the accounts page or a sub-page.

### Missing nav-agency-dashboard link

#### Severity
MEDIUM

#### Description
The story context mentions a `[data-test='nav-agency-dashboard']` placeholder hook from story 678. No such link exists anywhere in the rendered HTML on `/accounts` or other authenticated pages. Agency users need a navigation entry point to reach the dashboard. The link should be conditionally rendered for agency-typed accounts.

### qa_seeds.exs prints port 4007 in magic-link URL

#### Severity
LOW

#### Scope
QA

#### Description
The seed script at `priv/repo/qa_seeds.exs` hardcodes `http://localhost:4007` in the printed magic-link URL. The live dev server is running on port 4008 (4007 serves stale code returning 404 on `/auth/github`). The QA tester must manually change the port from 4007 to 4008 when using the printed URL. The seed script should use `Application.get_env(:market_my_spec, MarketMySpecWeb.Endpoint, [])[:url][:port]` or a configurable base URL rather than hardcoding port 4007.
