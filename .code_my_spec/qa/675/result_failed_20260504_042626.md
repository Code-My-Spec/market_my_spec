## Issues

### Anemic spex: 7 of 9 criteria unreachable without bearer token

#### Severity
HIGH

#### Scope
QA

#### Description
Criteria 5715, 5716, 5723, 5724, 5725, 5726, and 5729 all share the same pattern: a full OAuth PKCE scaffolding block (register → authorize LiveView → approve → exchange code → bearer) followed by one meaningful behavioral assertion. The scaffolding is copy-pasted verbatim across all 7 files with no shared helper. None use `Frame.execute`, `Anubis.Test`, or any unit-level approach — they drive real HTTP against a running server.

This means these tests cannot run in any environment that cannot complete the full OAuth PKCE flow (CI without a browser, sub-agent QA, or integration test suites that don't spin up LiveView). The 7 behavioral assertions buried inside them (SSE 200, JSON-RPC error for non-SSE read, SKILL.md content in response, unknown skill error, step 3 content isolation, not-found error for missing file, path-traversal rejection) are valuable, but none are reachable.

The path-traversal check (5729) is the highest-risk scenario to leave untested — it is the only runtime guard between a bearer-holding client and arbitrary filesystem reads.

Recommended fix: extract the OAuth scaffolding into a shared `Fixtures.mint_mcp_bearer/1` helper (or use `Anubis.Test` helpers if the library provides them), and allow the behavioral `when_/then_` steps to start from a pre-minted bearer. Alternatively, unit-test the tool/resource modules directly using `Frame` construction without HTTP.

### OAuth well-known metadata hardcodes port 4000 instead of runtime port

#### Severity
MEDIUM

#### Scope
APP

#### Description
Both `/.well-known/oauth-authorization-server` and `/.well-known/oauth-protected-resource` return URLs with `http://localhost:4000` hardcoded:

- `authorization_endpoint: http://localhost:4000/oauth/authorize`
- `token_endpoint: http://localhost:4000/oauth/token`
- `registration_endpoint: http://localhost:4000/oauth/register`
- `issuer: http://localhost:4000`
- `resource: http://localhost:4000/mcp` (protected-resource doc)
- `authorization_servers: ["http://localhost:4000"]` (protected-resource doc)

The MCP server is running on port 4008 in dev. A standards-compliant MCP client (e.g., Claude Code) that follows OAuth 2.0 Resource Server metadata discovery per RFC 8707 would attempt to contact `localhost:4000` for all OAuth operations, which is not the app.

The `OauthController` generating these responses should use `MarketMySpecWeb.Endpoint.url()` or equivalent runtime URL construction rather than a hardcoded base URL.

Reproduction: `curl http://localhost:4008/.well-known/oauth-authorization-server | jq .issuer` returns `"http://localhost:4000"`.

### SSE Accept header not recognized — 406 for standards-compliant SSE clients

#### Severity
MEDIUM

#### Scope
APP

#### Description
The `:mcp_authenticated` pipeline is configured with `plug :accepts, ["json", "sse"]`. Phoenix's MIME type alias `"sse"` maps to `application/sse+json`, but the SSE standard (EventSource API, MCP StreamableHTTP spec) uses `text/event-stream` as the `Accept` header. Sending `Accept: text/event-stream` to `GET /mcp` returns a `406 Not Acceptable` before authentication even runs.

Reproduction:
```
curl -i -X GET -H "Accept: text/event-stream" http://localhost:4008/mcp
HTTP/1.1 406 Not Acceptable
Expected one of ["json", "sse"] but got the following formats:
* "text/event-stream" with extensions: []
```

This means any conforming SSE client (browser EventSource, Claude Code MCP transport) that sends `Accept: text/event-stream` cannot establish a session. The `:accepts` plug needs to include the correct MIME type that maps to `text/event-stream`, or the custom MIME type `"sse"` needs to be registered in Phoenix to map to `text/event-stream`.

Fix: either add `"text/event-stream"` to Phoenix's MIME type registry via `config :mime, :types, %{"text/event-stream" => ["sse"]}` in `config.exs`, or verify that Anubis registers this type and ensure the registration happens before the Endpoint compiles.
