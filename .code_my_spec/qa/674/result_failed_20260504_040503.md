## Issues

### StartInterview MCP Tool Is a Stub — Story 674 Spec Wants Real Interview Behavior

#### Severity
HIGH

#### Scope
APP

#### Description
`MarketMySpec.McpServers.MarketingStrategy.Tools.StartInterview` returns `"Interview started for topic: #{topic}"` and nothing else. Story 674 acceptance criteria require the agent to load the marketing-strategy playbook on tool invocation, return SKILL.md content as orientation (the `SkillOrientation` resource exists at `marketing-strategy://orientation` but the tool never references it), and walk the user through the 8-step flow. The `@moduledoc` explicitly acknowledges this: "Story 674 will replace this stub with real interview logic." This means the primary deliverable of story 674 — functional interview behavior — has not been implemented. All behavioral acceptance criteria (5731 behavioral component, 5733–5737, 5738–5742) cannot be verified with a live exchange.

### Story 674 Spex Files Are Anemic — 11 of 12 Are Static SKILL.md String Checks, Not Behavioral MCP Exchange Tests

#### Severity
HIGH

#### Scope
QA

#### Description
11 of 12 spex criteria (5731, 5733, 5734, 5735, 5736, 5737, 5738, 5739, 5740, 5741, 5742) use `File.read!/1` against SKILL.md or a step file and assert substring presence with `=~`. These tests verify that the playbook document contains the right text, but they cannot detect whether the MCP server actually delivers that content to an agent, whether the agent honors the instructions, or whether any behavioral acceptance criteria are met.

The correct test structure for these criteria would be:
- `given_` step: mint a bearer token and establish an authenticated MCP session
- `when_` step: post a real JSON-RPC `tools/call` invocation via the conn or live server
- `then_` step: assert against the actual JSON-RPC response body

Currently, none of criteria 5731 and 5733–5742 can fail for behavioral reasons — they can only fail if someone edits SKILL.md to remove specific phrases. Criterion 5732 is the only substantive test: it makes a real MCP request and asserts HTTP 401 + WWW-Authenticate.

### SKILL.md Assertion String Mismatches Will Cause Spex Failures

#### Severity
MEDIUM

#### Scope
APP

#### Description
Four spex assertions do not match the actual SKILL.md content:

1. **Criterion 5731** asserts `steps/01_current_state.md` — SKILL.md file tree lists `01_current_state.md` without the `steps/` prefix. The `steps/` prefix appears in prose only. Fix: add `steps/` prefix to the tree entries, OR update the spex assertions to match the current format.

2. **Criterion 5731** asserts `steps/08_plan.md` — same issue, bare `08_plan.md` in the tree.

3. **Criteria 5737 and 5738** assert `Don't batch` (capital D) — SKILL.md uses `don't batch` (lowercase d, mid-sentence). Fix: add a sentence starting with `Don't batch` as a standalone imperative, OR update the spex.

4. **Criterion 5737** asserts `if the user bails after step 3` (lowercase i) — SKILL.md uses `If the user bails after step 3` (uppercase I, start of sentence). Fix: change the spex assertion to `If the user bails after step 3`, or add a lowercase variant.

### QA Plan Documents Wrong App Port

#### Severity
LOW

#### Scope
QA

#### Description
The QA plan at `.code_my_spec/qa/plan.md` documents the app base URL as `http://localhost:4007`. The running app responds on port **4008** (confirmed: `curl http://localhost:4008/mcp` returns 401, `curl http://localhost:4007/` returns 500). The plan's Tools Registry and Seed Strategy sections both reference port 4007. Any QA agent following the plan will probe the wrong port and get misleading results. Update the plan's base URL to `http://localhost:4008`.
