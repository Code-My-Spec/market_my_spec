fail

## Scenarios

### Scenario 1: Login page renders only the magic-link form (criterion 5683)

FAIL

Navigated to `http://localhost:4008/users/log-in`. The page rendered both `#login_form_magic` AND `#login_form_password`. The BDD spec asserts `refute has_element?(context.view, "#login_form_password")` — this assertion would fail against the running app.

`#login_form_magic` is present with an `input[type=email]` — the magic-link email form is rendered correctly. The "Log in with email" button is present. However, a second form `#login_form_password` with email + password fields and "Log in and stay logged in" / "Log in only this time" buttons is also rendered, separated by a horizontal divider "or".

The source code at `lib/market_my_spec_web/live/user_live/login.ex` lines 62-92 unconditionally renders `#login_form_password`. The story requires magic-link-only UI.

### Scenario 6: Direct POST to password endpoint is rejected (criterion 5684)

PASS

Tested via the browser `#login_form_password` form (which submits via native POST to `/users/log-in`). Submitted `qa@marketmyspec.test` + `wrongpassword`. Server responded with flash "Invalid email or password" and redirected back to `/users/log-in`. No new session was established by the password POST. The existing session (from prior magic-link sign-in) remained unchanged — confirming the password path does not authenticate.

Code review of `user_session_controller.ex` confirms: the email+password path calls `Users.get_user_by_email_and_password/2`. A magic-link-only user has no `hashed_password`, so this returns nil and the controller redirects with an error — no session token is issued.

## Issues

### Login page renders password form alongside magic-link form

#### Severity
HIGH

#### Scope
APP

#### Description
The login page at `/users/log-in` renders both `#login_form_magic` (the magic-link email form) and `#login_form_password` (an email + password form with "Log in and stay logged in" and "Log in only this time" buttons) separated by a horizontal "or" divider.

The story acceptance criterion "Login page renders only the magic-link form" and BDD spec criterion 5683 both require that `#login_form_password` is NOT present: `refute has_element?(context.view, "#login_form_password")`.

Reproduced at: `http://localhost:4008/users/log-in`

Source: `lib/market_my_spec_web/live/user_live/login.ex` lines 62-92 — the `#login_form_password` form block should be removed or gated behind a feature flag for this story to pass.

### QA seed script hardcodes port 4007 in printed URL

#### Severity
LOW

#### Scope
QA

#### Description
`priv/repo/qa_seeds.exs` line 47 prints the magic-link URL as `http://localhost:4007/users/log-in/<token>`. The current dev server runs on port 4008 (the 4007 server is stale and lacks current routes). The QA agent must manually substitute the correct port when using the printed URL.

Workaround used: replaced 4007 with 4008 in the URL before navigating. The token itself is valid regardless of port.

Fix: use `Application.get_env(:market_my_spec, MarketMySpecWeb.Endpoint)[:http][:port]` or a dynamic URL helper in the seed script output.
