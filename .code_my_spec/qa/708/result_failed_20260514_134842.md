## Issues

### Venues not persisted to database — LiveView uses in-memory socket assigns only

#### Severity
HIGH

#### Description
The `VenueLive.Index` LiveView manages venues exclusively in socket assigns (`@venues` list). All four operations — add, toggle enabled, delete — mutate the in-memory list only.

`handle_event("save_venue", ...)` creates a map `%{id: System.unique_integer([:positive]), source: ..., identifier: ..., weight: ..., enabled: true}` and appends it to `socket.assigns.venues`. It does NOT call `VenuesRepository.create_venue/2`.

Similarly, `handle_event("toggle_enabled", ...)` maps over the in-memory list, and `handle_event("delete_venue", ...)` filters the in-memory list.

As a result:
- All venues are lost on page navigation or LiveView disconnect
- The database `venues` table remains empty regardless of UI actions
- Story 705's search can never read venues because none are written to the DB
- The `VenuesRepository` module is fully implemented (`create_venue`, `list_venues`, `update_venue`, `delete_venue`) but is never called by the LiveView

Repro: Add a venue via the UI, navigate to `/accounts`, return to `/accounts/{id}/venues` — empty state is shown.

### Two app instances running on different ports with different code versions

#### Severity
MEDIUM

#### Scope
QA

#### Description
During testing, two instances of the market_my_spec app were observed running simultaneously:
- Port 4007: Has the current code including `/accounts/:id/venues` and `/accounts/:id/threads` routes
- Port 4008: Has older code without these routes (returns 404 for the venues URL)

The `priv/repo/qa_seeds.exs` script hardcodes `http://localhost:4008` in its output URLs regardless of the PORT environment variable used to run the seeds. This caused confusion: seed output showed 4008 URLs, but the actual app with venue support is on 4007.

The QA plan notes port 4007 as the dev server port (correct for the current-code instance), but the seeds output disagrees.

Workaround: Use port 4007 for all browser testing. Fix: update seeds script to use the app's configured endpoint URL (or `System.get_env("PORT", "4007")`), and stop the stale 4008 instance.

### Seeds script hardcodes port 4008 instead of using configured app URL

#### Severity
LOW

#### Scope
QA

#### Description
`priv/repo/qa_seeds.exs` prints magic-link URLs hardcoded to `http://localhost:4008`. The app runs on port 4007 in dev (as documented in the QA plan). The script should use `MarketMySpecWeb.Endpoint.url()` or reference `System.get_env("PORT", "4007")` so the printed URLs are correct.

When a tester uses the 4008 URL from seed output to sign in, the session is established at port 4008 — but the venues route (and other newer routes) only exist on 4007. After the Confirmation LiveView redirects back using the endpoint's configured URL (4007), the tester ends up with a session at 4007, but the experience is confusing.
