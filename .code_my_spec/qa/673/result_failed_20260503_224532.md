Navigated to `http://localhost:4008/users/log-in`. The page renders a "Sign in with Google" button (`data-test="google-sign-in"`) but there is no GitHub sign-in button on the page. The element `[data-test="github-sign-in"]` is absent.

### Scenario 2 — GitHub OAuth flow initiation (Criterion 5685 / public route)

FAIL (functional, but fails to reach GitHub due to module name typo)

`GET /auth/github` returns HTTP 302. The `UserOAuthController` catches the exception (the `rescue` block fires because `Providers.Github` does not exist — see Issue 2 below) and redirects to `/users/log-in` with flash: "Failed to connect to GitHub. Please try again." The route is wired and the error handling is graceful, but the flow does NOT redirect to GitHub as required.

### Scenario 5 — GitHub Connect from integrations page crashes (Criterion 5685 / related)

FAIL

Clicking the GitHub "Connect" button on `/integrations` triggers `GET /integrations/oauth/github`. The `IntegrationsController.redirect_uri/1` calls `Integrations.callback_path(:github)` which calls `MarketMySpec.Integrations.Providers.Github.callback_path/0`. This module does not exist (the real module is `MarketMySpec.Integrations.Providers.GitHub` with capital H in HUB). The IntegrationsController has no rescue block, so this crashes with a 500 `UndefinedFunctionError` in dev and would be an unhandled exception in prod.

Root cause: `config/config.exs` line 31 maps `github:` to `MarketMySpec.Integrations.Providers.Github` (lowercase `h`) when the actual module is `MarketMySpec.Integrations.Providers.GitHub` (uppercase `H`). The `integration_providers` config and `oauth_providers` config both use the `:github` atom, but the module alias differs.

Evidence: `.code_my_spec/qa/673/screenshots/github-connect-error.png`

## Issues

### GitHub sign-in button missing from login page

#### Severity
HIGH

#### Scope
APP

#### Description
The login page (`lib/market_my_spec_web/live/user_live/login.ex`) has a Google sign-in button (`data-test="google-sign-in"`) but no GitHub sign-in button. The BDD spec criterion 5685 asserts `has_element?(view, "[data-test='github-sign-in']")`. The public OAuth route `GET /auth/github` exists and is wired, but there is no UI entry point for it on the login page.

Fix: add a GitHub sign-in link to the login template, parallel to the Google button, with `href={~p"/auth/github"}` and `data-test="github-sign-in"`.

URL: `http://localhost:4008/users/log-in`

### Module name typo in config: `Providers.Github` should be `Providers.GitHub`

#### Severity
HIGH

#### Scope
APP

#### Description
`config/config.exs` line 31 maps the `:github` OAuth provider to `MarketMySpec.Integrations.Providers.Github`, but the actual module is `MarketMySpec.Integrations.Providers.GitHub` (capital `H` in `HUB`). This typo has two visible effects:

1. `GET /integrations/oauth/github` crashes with `UndefinedFunctionError: function MarketMySpec.Integrations.Providers.Github.callback_path/0 is undefined`. The `IntegrationsController` has no rescue/catch, so this surfaces as a 500 in dev and an unhandled crash in prod.

2. `GET /auth/github` silently fails — the `UserOAuthController` rescues the same exception and flashes "Failed to connect to GitHub. Please try again." rather than redirecting to GitHub. This means GitHub sign-up/sign-in via the public auth route is completely broken.

Fix: change line 31 of `config/config.exs` from:
```
github: MarketMySpec.Integrations.Providers.Github
```
to:
```
github: MarketMySpec.Integrations.Providers.GitHub
```

This single character fix unblocks both the public sign-in flow and the integrations connect flow.

Reproduction:
1. Visit `http://localhost:4008/auth/github` — redirected to login with "Failed to connect to GitHub" flash
2. Sign in, visit `/integrations`, click GitHub Connect — 500 UndefinedFunctionError

### QA plan port note needs updating

#### Severity
LOW

#### Scope
QA

#### Description
The QA plan documents port 4007 as the base URL, but the active server with current code is on port 4008. The seed script prints magic-link URLs with port 4007. QA testers must manually substitute port 4008 in magic-link URLs and curl commands. This creates friction and potential for errors.

The plan's System Issues section documents the dotenvy/PORT issue. The active server port should be verified at the start of each QA session with a quick probe (`curl -o /dev/null -w "%{http_code}" http://localhost:4007/auth/github` — if 404, use 4008).
