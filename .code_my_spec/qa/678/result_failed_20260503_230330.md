fail

The page shows "You don't have any accounts yet." — no default individual account is created on sign-up. The user must manually create an account.

### Scenario 6: Account creator is automatically the owner (criterion 5768)

PARTIAL PASS

The `/accounts/:id/members` page correctly shows the creator with the "Owner" role badge. However, the accounts list at `/accounts` does NOT display the role next to each account card. The BDD spec asserts `html =~ ~r/Owner Test Workspace[\s\S]{0,400}\bowner\b/i` against the accounts list — this would fail because the accounts list HTML contains no role label.

Screenshot (members page shows owner): `.code_my_spec/qa/678/screenshots/678-account-members.png`
Screenshot (accounts list lacks role): `.code_my_spec/qa/678/screenshots/678-accounts-after-create.png`

### Scenario 7: Self-service form has no type selector (criterion 5777)

PASS (partial — no `data-test` attributes)

The `/accounts/new` form has no `account[type]` selector. There is no way for a self-service user to select account type. However, the BDD spec relies on `data-test="account-form"` selector and `data-test="account-type-individual"` / `data-test="account-type-agency"` attributes, none of which are present in the rendered HTML. The form uses `id="account-form"` (not `data-test`), and no account-type markers exist in the accounts list.

Screenshot: `.code_my_spec/qa/678/screenshots/678-accounts-new.png`

### Scenario 8: Individual account does not show agency features (criterion 5775)

PASS (by absence)

The accounts list and account management pages contain no "manage clients" or "agency dashboard" affordances. No agency-only features are exposed for self-service created accounts.

However, there is no `type` field in the Account schema or database — accounts are neither marked "individual" nor "agency". The absence of agency features is structural, not intentional typing.

### Scenario 9: Agency account features / admin-provisioned agency (criteria 5776, 5778)

UNTESTABLE

The Account schema has no `type` field (no `individual`/`agency` column in the database or Ecto schema). No `Fixtures.agency_account_fixture/1` exists. Agency account features cannot be tested.

However, the container has no `data-test="account-picker"` attribute — the BDD spec asserts `has_element?(view, "[data-test='account-picker']")` which would fail.

Screenshot: `.code_my_spec/qa/678/screenshots/678-picker-multiple-accounts.png`

## Issues

### No default individual account created on sign-up
#### Severity
HIGH
#### Description
New users complete the registration + confirmation flow and land with zero accounts. The acceptance criterion requires a default individual account to be created automatically on sign-up. No account creation happens — the user sees "You don't have any accounts yet." on `/accounts`.

URL: `http://localhost:4008/accounts` after a fresh sign-up.

### User with no accounts not redirected to account creation
#### Severity
HIGH
#### Description
An authenticated user with zero accounts can freely access protected routes like `/users/settings` without being redirected to `/accounts/new`. The acceptance criterion requires this redirect to fire for any protected route when the user has no account memberships.

Confirmed by visiting `http://localhost:4008/users/settings` as a freshly registered user with no accounts.

Screenshot: `678-settings-no-redirect.png`

### New user not sent to account creation after sign-up
#### Severity
HIGH
#### Description
After confirming their account via magic link, new users are redirected to the home page (`/`) instead of `/accounts/new`. The acceptance criterion states new users must be directed to explicit account creation before reaching any dashboard.

Screenshot: `678-new-user-post-confirm.png`

### Accounts list does not display user role badge per account
#### Severity
MEDIUM
#### Description
The `/accounts` index page renders account cards without displaying the current user's role in each account. The BDD spec for criterion 5768 asserts that the string "owner" appears near the account name in the accounts list HTML. It does not. Role is only visible on the `/accounts/:id/members` sub-page.

Affected assertion: `html =~ ~r/Owner Test Workspace[\s\S]{0,400}\bowner\b/i`

### No account type field (individual/agency) in schema or database
#### Severity
HIGH
#### Description
The `accounts` table and `Account` Ecto schema have no `type` field. There is no `individual` or `agency` distinction anywhere in the data layer. Multiple acceptance criteria depend on this distinction (5766, 5775, 5776, 5777, 5778). The accounts list shows no `data-test="account-type-individual"` or `data-test="account-type-agency"` markers. Agency-related criteria (5776, 5778) are entirely untestable.

### Missing data-test attributes on account UI elements
#### Severity
MEDIUM
#### Description
BDD specs for story 678 rely on `data-test` CSS selectors that do not exist in the rendered HTML:
- `[data-test='account-form']` — form has `id="account-form"` but not `data-test`
- `[data-test='account-row']` — no such attribute on account cards
- `[data-test='account-type-individual']` and `[data-test='account-type-agency']` — not present
- `[data-test='account-picker']` — not present on the picker container
- `[data-test='nav-agency-dashboard']` — not present

All BDD spex tests will fail at the selector level regardless of behavioral correctness.

### Inviting an existing account member shows no error to user
#### Severity
MEDIUM
#### Description
When a user who is already a member of an account is invited again via the invitations form, the server correctly rejects the request (the `validate_user_not_already_member/2` function exists in the Accounts context), but the UI shows no error message — the invite form stays open silently. No flash message or inline form error is displayed.

Reproduced: invited `brand-new-user-678@marketmyspec.test` after they had accepted the first invitation and were a confirmed member.

Screenshot: `678-add-existing-member.png`

### Duplicate pending invitations to same email are not blocked
#### Severity
MEDIUM
#### Description
Submitting the invite form twice for the same email address (before the invitee has accepted) creates two separate pending invitations in the database. The invitations list shows both rows with no error or deduplication. The `invite_user/4` function in the Accounts context validates against existing *members* but not against existing *pending invitations*.

Screenshot: `678-duplicate-invitation.png`

### Invitation email links use localhost:4000 instead of app port
#### Severity
LOW
#### Scope
QA
#### Description
Invitation emails generated by `InvitationNotifier` use `http://localhost:4000` as the base URL (hardcoded in `Application.get_env(:market_my_spec, :base_url, "http://localhost:4000")`). The dev server runs on port 4008. Testers must manually replace the port in the link before following it. Fix: set `base_url` in `config/dev.exs` or `envs/dev.env` to match the running port.

Location: `lib/market_my_spec/accounts.ex`, `defp send_invitation_email/1`.
