## Issues

### FilesLive.Show exposes raw Elixir error atom to browser

#### Severity
MEDIUM

#### Scope
APP

#### Description
Navigating to any `/files/*key` URL (e.g., `http://localhost:4008/files/marketing/05_positioning.md`) renders the raw Elixir error value `:no_active_account` in red in the page body. The Show LiveView uses `inspect(reason)` to format the error, which is appropriate during development but exposes internal implementation details to the user.

The root cause is that `Scope.for_user(user)` sets `active_account_id: nil`, and the Files context returns `{:error, :no_active_account}` rather than fetching the file. The Show LiveView then renders `:no_active_account` directly.

A user-friendly message should replace the raw atom, e.g., "No account selected" or a redirect to the account picker.

### active_account_id is never populated in web-session Scope

#### Severity
HIGH

#### Scope
APP

#### Description
The `Scope.for_user/1` function always sets `active_account_id: nil`. Nothing in the authentication pipeline, the `:require_account_membership` on_mount hook, or the FilesLive mounts sets `active_account_id` to the user's actual account ID.

As a result, `Files.list/2` and `Files.get/2` always return `{:error, :no_active_account}` for every web session, regardless of account membership. The `/files` index silently swallows this error and shows "No artifacts yet." The `/files/*key` show page renders the raw atom.

This means the Files UI cannot display any artifacts written by the MCP agent until the session scope is wired to carry the active account ID. The story 676 end-to-end acceptance criterion ("user completes step 5 and finds positioning.md in their account workspace") cannot be satisfied via the UI without this fix.

### MCP tool modules WriteFile, ReadFile, ListFiles not implemented

#### Severity
HIGH

#### Scope
APP

#### Description
Criteria 5849 and 5850 require `MarketMySpec.McpServers.Marketing.Tools.{WriteFile, ReadFile, ListFiles}` but none of these modules exist. The tools directory (`lib/market_my_spec/mcp_servers/marketing_strategy/tools/`) contains only `start_interview.ex`.

Additionally, there is a namespace mismatch: the existing module uses `MarketingStrategy.Tools` but the spex reference `Marketing.Tools`. One of these needs to align.

The `account_scoped_user_fixture/0` in `test/support/fixtures/users_fixtures.ex` also sets `active_account_id: nil` via `Scope.for_user(user)`, which would cause any `WriteFile.execute/2` call to fail with `:no_active_account` even once the module is implemented. The fixture should set `active_account_id` to the user's created account ID.

### account_scoped_user_fixture creates scope with nil active_account_id

#### Severity
MEDIUM

#### Scope
APP

#### Description
`account_scoped_user_fixture/0` in `test/support/fixtures/users_fixtures.ex` calls `user_fixture()` (which creates a default individual account) but then returns `Scope.for_user(user)`, which always has `active_account_id: nil`.

Spex 5849 and 5850 depend on this fixture to produce a scope that can write and read files. Because `active_account_id` is nil, the Files context returns `{:error, :no_active_account}` at the first `WriteFile.execute/2` call.

The fixture needs to fetch the user's default account and include `active_account_id` in the returned Scope — or `Scope.for_user/1` needs to be extended to look up and populate the default active account.
