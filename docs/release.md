# MMS Agent Release Process

How to ship a new version of the MMS Agent binary via GitHub Releases + Homebrew.

## One-time bootstrap (do this once, before the first release)

### 1. Create the Homebrew tap repo

The tap repo holds the Homebrew formula. Naming convention requires `homebrew-<tap_name>`.

```
GitHub repo: Code-My-Spec/homebrew-mms-agent  (or wherever you own)
  └── Formula/
      └── mms-agent.rb    ← auto-generated by .github/workflows/release.yml
```

Create the empty repo on GitHub. No README needed; the release workflow will populate `Formula/mms-agent.rb` on every tag push.

### 2. Provision a token for the release workflow to push to the tap repo

The release workflow needs to write to the tap repo from the `market_my_spec` repo. The built-in `GITHUB_TOKEN` cannot push cross-repo, so create a fine-grained personal access token:

1. <https://github.com/settings/personal-access-tokens/new>
2. Repository access: only `Code-My-Spec/homebrew-mms-agent`
3. Permissions:
   - Contents: Read and write
   - Metadata: Read-only (auto)
4. Generate, copy the token

Add it as a repository secret on `Code-My-Spec/market_my_spec`:

```
Settings → Secrets and variables → Actions → New repository secret
Name:  HOMEBREW_TAP_TOKEN
Value: <the token>
```

### 3. Push the initial formula (optional)

The CI workflow will create the formula on the first tag push, but if you want a placeholder that lets users `brew tap` before the first release, copy `homebrew/Formula/mms-agent.rb` from this repo into the tap repo manually and commit it.

## Per-release flow

### Cut a release

```bash
just release 0.2.0
```

This bumps `mix.exs` and `config/config.exs` (the `agent_latest_version`), commits the bump, tags `v0.2.0`, and pushes both the commit and the tag.

### What CI does on tag push

`.github/workflows/release.yml` triggers on `v*` tags and:

1. **Builds in parallel on four runners** (macos-14, macos-13, ubuntu-22.04, ubuntu-22.04-arm) — one Burrito target each, using Zig 0.15.2 + Elixir 1.19.4 / OTP 28.1.
2. **Computes SHA256** for each binary.
3. **Creates a GitHub Release** named `MMS Agent v0.2.0` with all four binaries + a `SHA256SUMS.txt` manifest as assets.
4. **Updates `Formula/mms-agent.rb`** in `Code-My-Spec/homebrew-mms-agent` with the new version + checksums and pushes the commit.

### Deploy the server

After the GitHub Release is up, deploy `market_my_spec` so the bumped `:agent_latest_version` is live on the server. The binary's phone-home check (`GET /api/agent/version`) needs to see the new version before it'll prompt users to upgrade.

```bash
# Whatever your deploy invocation is — kamal deploy, mix deploy, etc.
```

### User-side update

Users run:

```bash
brew upgrade mms-agent
```

The next time they invoke `mms-agent pair` or `mms-agent status`, the version-check sees they're current and doesn't print anything. Users on older versions see:

```
Notice: mms-agent v0.2.0 is available. Run: brew upgrade mms-agent
```

## First-time user install

```bash
brew tap Code-My-Spec/mms-agent       # adds the tap
brew install mms-agent                # downloads the binary for their platform
mms-agent pair                        # opens browser, completes pairing
mms-agent server                      # stays running, joins the channel
```

## How the auto-update works

- The binary's compiled-in version comes from `Application.spec(:market_my_spec, :vsn)` (whatever was in `mix.exs` at build time).
- On every CLI invocation that isn't `server` or `help`, the binary calls `GET <server_url>/api/agent/version` (cached to `~/.mms-agent/version_check.json` with a 24h TTL).
- If `latest > current`, it prints the upgrade notice to stderr and continues with the user's command.
- Homebrew handles the actual download, checksum verification, PATH swap, and Gatekeeper quarantine clearance.
- No in-binary self-replace, no Apple Developer ID requirement.

## Signing notes (deferred until needed)

The first release ships unsigned. macOS users installing via Homebrew get quarantine cleared automatically by `brew install`. Users who download the binary directly from GitHub Releases would need to run:

```bash
xattr -d com.apple.quarantine market_my_spec_agent_macos_m1
```

When you're ready to sign + notarize ($99/yr Apple Developer Program), see `.code_my_spec/knowledge/devops/signing-and-gatekeeper.md` for the workflow integration.

## Rolling back a bad release

GitHub Releases supports delete + retag. The Homebrew formula needs to be reverted manually in the tap repo (push a commit that points the URLs and SHA256s back at the previous version). Users on the bad version will need to `brew upgrade` once the tap is fixed.

For now there's no in-server kill switch (`min_supported_version` is read but never enforced by the binary). If a release is catastrophic, file an issue to add an enforcement check.
